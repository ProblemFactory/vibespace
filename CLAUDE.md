# Claude Code WebUI

## Project Overview
A web-based UI for managing multiple Claude Code sessions with a tiling window manager, file explorer, and code editor. Uses native `claude` CLI via **dtach** for session persistence across server restarts.

## How to Build and Run
```bash
# One-line install (checks deps, clones, builds):
curl -fsSL https://raw.githubusercontent.com/ProblemFactory/claude-code-webui/master/install.sh | bash

# Or manually:
npm install
npm run build   # esbuild bundles src/client.js Ôćĺ public/bundle.js (~1.8MB)
node server.js  # starts on port 3456 (or PORT env var)
```

## Architecture

### Stack
- **Backend**: Node.js + Express + WebSocket (`ws`) + `node-pty` + dtach
- **Frontend**: Vanilla JS (ES modules) bundled with esbuild, xterm.js for terminals, CodeMirror 6 for editor
- **Session persistence**: dtach ÔÇö claude processes run in dtach sessions, survive server restarts
- **Data persistence**: `data/layouts.json`, `data/session-meta/*.json`, `data/sockets/cw-*`


### Architecture Patterns

- **All frontend cross-class communication** goes through `App` (mediator pattern). Classes receive `app` in constructor and call `app.methodName()`.
- **One-time WebSocket handler pattern**: Register via `ws.onGlobal()`, match on `type` + `sessionId`, remove self after first match. Used in `createSession()`, `attachSession()`, `resumeSession()`.
- **Debounced auto-save**: `LayoutManager.scheduleAutoSave()` waits 2s after last change. Blocked by `_restoring` flag for 5s on page load.
- **Resizer**: Reusable component with `inside: true` mode for fixed-position elements (sidebar). Don't set `position` or `flex` on elements that aren't flex children.
- **Layout restore**: `attachSession()` returns `winInfo` synchronously (the DOM element). Position is applied directly to this `winInfo`, NOT by guessing via `windows.values().pop()`. The WebSocket attach is async but the window element already exists.
- **Proportional bounds tracking**: `win.gridBounds` stores position `{left, top, width, height}` as fractions (0-1) of workspace. `_reflowWindows()` (via ResizeObserver) recalculates pixel positions on workspace resize. Applies universally ÔÇö grid snap, edge snap, drag-drop, and `applyLayout()` all capture bounds. User resize updates proportions on mouseup. Works in both grid and freeform modes.
- **Overlap switcher**: Right-click on title bar detects overlapping windows via rect intersection (`_rectsOverlap`), shows popup at cursor position. Works in any mode ÔÇö not tied to grid cells.

## Key Design Decisions & Lessons Learned

### 1. dtach for Session Persistence (migrated from tmux)
**Decision**: Claude processes run in dtach sessions, NOT tmux or direct children of Node.js.

**Why dtach over tmux**: dtach is a minimal PTY detach/attach tool with ZERO rendering layer. tmux interposes its own terminal rendering that intercepts mouse events, scroll, escape sequences. After 6+ iterations of tmux mouse config (mouse on/off, send-keys -M, allow-passthrough, conditional bindings, stripping mouse sequences client-side), none worked correctly. dtach passes all PTY I/O transparently.

**Implementation**:
- Create: `dtach -c <socket> -E node pty-wrapper.js <buf> <meta> env EDITOR=... claude ...` (create+attach, with PTY wrapper for output buffering)
- Attach: `pty.spawn('dtach', ['-a', socketPath, '-E', '-r', 'winch'])` (SIGWINCH on attach)
- Server restart: scan `data/sockets/cw-*`, re-attach to surviving sessions, read buffer from wrapper's `.buf` file
- Metadata in `data/session-meta/<sockName>.json`
- Output buffer in `data/session-buffers/<sessionId>.buf` (written by pty-wrapper, survives server restart)

**PTY Wrapper** (`data/bin/pty-wrapper.js`): Runs inside dtach, uses node-pty to spawn claude. Tees all output to stdout (dtach PTY) + buffer file. Survives server restarts ÔÇö dtach keeps the wrapper alive, wrapper keeps writing the buffer. On server restart, `restoreSessions()` reads the buffer file to recover terminal content.

**Why dtach -c not dtach -n**: `dtach -n` creates a detached session ÔÇö initial output is LOST because dtach doesn't buffer when no one is attached. `dtach -c` creates AND immediately attaches, so output flows through from the start.

**CRITICAL**: Never delete dtach socket files during server restart. The socket IS the session.

### 2. Mouse/Scroll Handling
With dtach, there is NO middle layer ÔÇö xterm.js talks directly to Claude Code's PTY. Mouse events, scroll, selection all work natively without any configuration.

**Focus event filtering**: xterm.js sends `\e[I` (focus in) and `\e[O` (focus out) to the PTY when the terminal element gains/loses browser focus. In split-pane mode (Ctrl+G), clicking between the editor and terminal causes rapid focus/blur events Ôćĺ `^[[I^[[O` spam in the terminal. Fix: strip `\e[I` and `\e[O` from `terminal.onData()` before sending to PTY.

**Previous tmux failures** (for reference, do NOT retry):
1. ÔŁî `mouse off` + strip sequences Ôćĺ no scroll
2. ÔŁî `mouse off` + wheelÔćĺarrow keys Ôćĺ arrows = input history, not scroll
3. ÔŁî `mouse on` + `send-keys -M` Ôćĺ breaks selection
4. ÔŁî Strip `\e[?1000h` client-side Ôćĺ same as mouse off
5. ÔŁî `mouse on` + conditional binding Ôćĺ Claude doesn't set mouse_any_flag
6. ÔŁî `mouse on` + various unbind combos Ôćĺ selection still broken

### 3. Ctrl+G External Editor
**Problem**: Claude Code has a hardcoded GUI editor list: `["code","cursor","windsurf","codium"]`. If EDITOR matches, it does NOT clear the screen. Otherwise it calls `enterAlternateScreen()` (clears screen).

**Solution**: Create a fake `code` script at `data/bin/code`. Set `EDITOR=data/bin/code`. Script communicates via HTTP (POST `/api/editor/open`), not terminal I/O. Claude Code sees "code" Ôćĺ GUI editor Ôćĺ no screen clear Ôćĺ shows "Save and close editor to continue...".

**Flow**: Ctrl+G Ôćĺ Claude invokes `code -w <tmpfile>` Ôćĺ our script POSTs to server Ôćĺ server broadcasts `editor-open` via WebSocket Ôćĺ client splits terminal window (top=terminal, bottom=CodeMirror) Ôćĺ "Save & Close" button Ôćĺ writes file + creates signal file Ôćĺ script exits Ôćĺ Claude reads file.

**DO NOT**: Print anything to terminal from editor helper. DO NOT filter `\e[?1049h` sequences client-side (breaks TUI rendering). The `data/bin/code` script is regenerated by `createEditorHelper()` in server.js on every startup.

### 4. Session Discovery (redesigned 2026-03-24)
**Lock-first approach** ÔÇö lock files are the source of truth for RUNNING, JONLs for HISTORY.

**Claude Code's internal session model:**
- Lock file: `~/.claude/sessions/<PID>.json` Ôćĺ `{pid, sessionId, cwd, startedAt}`
  - Created on start, deleted on graceful exit
  - `sessionId` changes on each `--resume` (new UUID, but JSONL filename stays original)
- JSONL file: `~/.claude/projects/<encodedCwd>/<originalSessionId>.jsonl`
  - Filename = original sessionId (immutable, even after multiple `--resume`)
  - Project dir encoding: `cwd.replace(/[/._]/g, '-')` (e.g. `/home/user/.config` Ôćĺ `-home-user--config`)

**Discovery algorithm:**
1. Scan lock files Ôćĺ filter alive PIDs Ôćĺ verify process is actually `claude` (avoids PID reuse)
2. Also scan tmux panes to determine if running claude is inside tmux (attachable)
3. For each alive lock, derive project dir from `lock.cwd` using the encoding formula
4. In that project dir, find the most recently modified JSONL Ôćĺ assign it as the running session
5. Each lock claims at most ONE JSONL (most recent in its project dir)
6. All unclaimed JONLs Ôćĺ STOPPED
7. Status values: `tmux` (in tmux, attachable), `external` (unknown terminal, not attachable), `stopped`

**Why previous approaches failed:**
- ÔŁî Match lock to JSONL by sessionId: fails after `--resume` (new lock ID, old JSONL filename)
- ÔŁî Match by cwd from inside JSONL: JSONL stores original cwd which may be stale after a directory move
- ÔŁî Match by cwd without dedup: one alive lock matches ALL JONLs in same directory (32 "running" for 10 PIDs)
- ÔŁî `recoverCwdFromProjDir()` greedy filesystem check: slow, fragile, wrong for moved directories

**Sidebar state machine:**
- **LIVE**: WebUI-managed (dtach), click to focus existing window
- **TMUX**: Running in external tmux, click to view (spawns `tmux attach -t <target>`). Closing window only detaches, does NOT kill the session.
- **EXTERNAL**: Running in unknown terminal, non-clickable
- **STOPPED**: Click to resume via `claude --resume`

### 5. Project Path Recovery
Claude encodes: `cwd.replace(/[/._]/g, '-')`. Forward direction (cwd Ôćĺ dir name) is deterministic.
Reverse direction (dir name Ôćĺ cwd) is ambiguous because `-` could be `/`, `.`, `_`, or literal `-`.
Recovery uses greedy filesystem checking as fallback, but the forward encoding is preferred.

### 6. Layout Persistence
- Auto-save: 2s debounce after window changes, saves to `data/layouts.json`
- Restore: always uses autoSave on page load
- `_restoring` flag blocks auto-save for 5s during restore (prevents empty state overwriting)
- Captures `claudeSessionId` per terminal for cross-restart matching
- Grid state (`{rows, cols}`) persisted and restored ÔÇö `setGrid()` triggers `_notify()` to save
- Custom grid presets stored in `layouts.json` Ôćĺ `customGrids` array, loaded on startup
- Window restore uses `winInfo` returned by `attachSession()` ÔÇö NOT `windows.values().pop()` (see bug below)
- **All window types restored**: terminals (by claudeSessionId match), file explorers (by explorerPath), file viewers/editors (by filePath), browser windows (by URL)

### 6a. Grid Presets
- **Freeform button**: Clears grid, restores free-floating mode (`setGrid(null)`)
- **Built-in presets**: maximize (1├Ś1), 2-col, 2-row, quad, 3-col ÔÇö hardcoded in `applyLayout()`
- **Custom grid presets**: "+" button opens dialog, saves to server via `POST /api/custom-grids`, renders button with auto-generated SVG icon (ÔëĄ4├Ś4) or "RxC" text (>4├Ś4). Right-click to delete.
- Custom grids use `grid-R-C` layout names (e.g. `grid-3-3`), parsed in `applyLayout()`
- `#custom-grids-container` uses `display: contents` to flatten into parent flex layout
- **All positioned windows track proportional bounds** via `win.gridBounds` `{left, top, width, height}` as fractions (0-1) of workspace. Captured after every positioning event: grid snap, edge snap, drag-drop, `applyLayout()`, user resize. `_reflowWindows()` (ResizeObserver on workspace) recalculates pixel positions on any workspace size change. Works in both grid and freeform modes.
- `gridBounds` is persisted in layout autoSave ÔÇö on restore, positioned via `_applyGridBounds()` instead of absolute pixels, adapting to current workspace size
- **Timing**: `_captureGridBounds` must run AFTER snap animation completes (220ms). Use `setTimeout(..., 250)`, not `10`.

### 7. Modular Frontend (refactored 2026-03-23)
Split from monolithic 1647-line `src/client.js` into 13 ES modules under `src/lib/`. All cross-class communication goes through the `App` mediator. esbuild follows ES imports automatically, no build config changes needed.

### 8. xterm.js Rendering Gotchas
- **Font change requires `clearTextureAtlas()`**: xterm.js canvas/WebGL renderer caches font glyphs in a texture atlas. Setting `terminal.options.fontFamily` alone does NOT re-render ÔÇö must call `terminal.clearTextureAtlas()` after changing font family or font size for the change to take visual effect, then call `fit()` to recalculate dimensions.
- **Focus events**: When focus reporting is enabled (`\e[?1004h`), xterm.js sends `\e[I`/`\e[O` on focus/blur. These must be filtered from `onData()` to prevent `^[[I^[[O` spam in split-pane UI.
- **Cursor**: Set `cursorBlink: false, cursorStyle: 'bar', cursorInactiveStyle: 'none'` ÔÇö Claude Code TUI renders its own visual cursor, so xterm.js's hardware cursor should be minimized.
- **Pin-to-bottom / scroll freeze**: When user scrolls up (detected via `wheel` event, not xterm `onScroll` which fires on programmatic scrolls too), output is queued in `_pendingOutput` instead of written to terminal. Terminal freezes in place. When user scrolls back to bottom or clicks the Ôćô button, `_repin()` writes all queued output and scrolls to bottom. This prevents Claude Code's TUI redraw (`\e[H` cursor home) from yanking the viewport.
- **Idle detection via OSC 0**: Claude Code updates terminal title via `\e]0;...\a`. First character indicates state: `Ôť│` (U+2733) = idle/waiting for input, braille chars (U+2800-28FF) = working. Detected via `terminal.parser.registerOscHandler(0, ...)`. When transitioning to idle while window not focused, `window-waiting` class triggers orange blink animation on title bar + taskbar item. Cleared on focus.

### 9. Usage / Rate Limit Monitoring
**Approach**: Minimal haiku API call every 5 minutes. Reads rate limit utilization from response headers (`anthropic-ratelimit-unified-*`). Uses OAuth token from `~/.claude/.credentials.json`.

**Available data**: 5-hour utilization (%), 7-day utilization (%), reset timestamps, status (allowed/limited).

**Not available via API**: "Sonnet only" weekly limit (only returned by internal `/api/oauth/usage` endpoint which has severe rate limiting). Claude Code gets this via `GET /api/oauth/usage` with `Bearer` + `anthropic-beta: oauth-2025-04-20` headers, but the endpoint is too rate-limited for periodic polling.

**Failed approaches:**
- ÔŁî Parse statusline output: user-customizable, can't guarantee format
- ÔŁî Statusline hook wrapper (`--settings`): intrusive, doesn't work for non-WebUI sessions
- ÔŁî `/api/oauth/usage` endpoint: correct data but extremely aggressive rate limiting
- Ôťů Haiku API call + response headers: reliable, cheap, gets 5h + 7d unified limits

### 10. Settings System (Global + Per-Terminal)
**Global settings** (toolbar ÔÜÖ): Theme, font size (A-/A+), font family. Stored in `localStorage` (`termFontSize`, `termFontFamily`). Changes propagate to all terminals without per-terminal overrides.

**Per-terminal settings** (window titlebar ÔÜÖ): Same three options, each with a "Default" that follows global. Theme and font have Default as a dropdown option. Font size has a Default checkbox that disables the number input. Overrides stored in `TerminalSession.overrides` and persisted in layout.

### 11. Font Discovery
**Priority**: Client-side `queryLocalFonts()` API (Chrome 103+) Ôćĺ server-side `fc-list :spacing=mono` fallback Ôćĺ always includes Google Web Fonts.

**Web Fonts** (via Google Fonts CSS in index.html): Fira Code, JetBrains Mono, Source Code Pro, IBM Plex Mono, Inconsolata. Always available regardless of local installs.

**Client-side detection**: `queryLocalFonts()` returns ALL local fonts. Monospace filtering via canvas measurement (`'i'` vs `'M'` width comparison). Requires user permission prompt (one-time).

**Server-side fallback**: `GET /api/fonts` runs `fc-list :spacing=mono family`, filters emoji/sign-writing fonts, caches result. Used when `queryLocalFonts()` is unavailable (Firefox, Safari, or denied permission).

**Font list structure**: Web fonts first, separator, local system fonts, "System Default" last.

**Why client fonts matter**: Terminal renders in the browser ÔÇö the fonts on the client machine are what determine rendering, not the server.

### Bug Fixes Applied
- Focus event spam (`^[[I^[[O`): stripped from terminal input
- Font changes: `clearTextureAtlas()` required
- Ctrl+G screen clearing: editor script named `code` (GUI editor whitelist)
- Session resume duplicates: `broadcastActiveSessions()` includes `claudeSessionId`
- Layout restore race: `_restoring` flag blocks auto-save
- Resumed sessions not showing: lock-first discovery with project dir matching
- Save & Close scroll: `scrollToBottom()` + `focus()` after editor close
- Sidebar resizer: `inside` mode for fixed-position elements
- Phantom cursor: disabled xterm.js cursor blink
- Session discovery inflation: `claimedLockPids` prevents one PID matching multiple JONLs
- Font switching no effect: original hardcoded fonts (Cascadia/SF Mono) not installed Ôćĺ Google Web Fonts + dynamic detection
- Web fonts not in dropdown: `document.fonts.check()` fails for unloaded CSS fonts Ôćĺ force-load via `document.fonts.load()`, then switched to `queryLocalFonts()` + server fallback approach
- Layout restore crash: removed `#theme-select` from HTML but `restoreState` still referenced it Ôćĺ TypeError crashed entire restore flow (grid, windows, sidebar all failed to restore)
- Layout restore wrong positions: `applyPosition` used `windows.values().pop()` to guess which window to position ÔÇö completely unreliable with async `attachSession`. Fix: `attachSession()` returns `winInfo`, restore applies position directly to it
- Grid state not persisted: `setGrid()` didn't call `_notify()`, so grid changes didn't trigger autoSave
- Custom grid buttons wrapping: `<span>` container didn't participate in flex layout Ôćĺ `display: contents` makes children flatten into parent flex
- Windows don't follow workspace resize: sidebar toggle or browser resize changed workspace but windows kept old pixel positions Ôćĺ `win.gridBounds` (proportional 0-1) + ResizeObserver calls `_reflowWindows()`. All positioning events (grid snap, edge snap, drag, applyLayout, user resize) capture bounds. Works in grid and freeform modes.
- applyLayout bounds not captured: `_captureGridBounds` ran at 10ms but snap animation takes 220ms Ôćĺ window still at old position when captured. Fix: delay to 250ms.
- Terminal bell notification: `terminal.onBell()` Ôćĺ show ­čöö on title when window not focused, clear on focus/click
- File explorer title: shows current path with front-truncation (`ÔÇŽ/long/path`) via `wm.setTitle()` on each `navigate()`
- White screen on resume: `dtach -n` loses initial output (no buffering when detached) Ôćĺ switched to `dtach -c` (create+attach). Also added `pty-wrapper.js` inside dtach for persistent buffering.
- Output lost on server restart: server-side buffer only exists in memory Ôćĺ `pty-wrapper.js` runs inside dtach independently, writes buffer to file continuously, survives server restarts
- Session kill leaves orphans: old kill only killed `dtach -a` (attach PTY), not `dtach -n` (session) Ôćĺ now uses `pgrep -f <socketPath>` to find and SIGTERM the dtach session process
- WebUI dtach sessions show as EXTERNAL: `/api/sessions` didn't check if PID was a webui-managed dtach child Ôćĺ added `webuiPids` detection via `pgrep -f <socketPath>` + recursive `ps --ppid` to find all descendants (dtach Ôćĺ pty-wrapper Ôćĺ claude)
- Named layout restore kills sessions: `loadNamed()` used `closeWindow()` which triggers `onClose` Ôćĺ kill. Fix: detach UI only (dispose terminal + remove DOM), don't send kill message. Sessions stay alive for re-attach.
- Font switching no effect: original hardcoded fonts (Cascadia/SF Mono) not installed Ôćĺ Google Web Fonts + dynamic detection
- Clipboard image paste failures: (1) `paste` event never fires ÔÇö xterm.js v5 uses Clipboard API directly, bypassing paste event. Fix: `attachCustomKeyEventHandler` intercepts Ctrl+V, redirects focus to hidden contenteditable div. (2) `navigator.clipboard.read()` unavailable over HTTP. Fix: contenteditable div receives real paste event with clipboardData.items. (3) `xclip -i file` unreliable. Fix: `cat file | xclip` pipe. (4) Bracketed paste `\e[200~...\e[201~` inserts text, doesn't trigger image check. Fix: send raw Ctrl+V (`\x16`) which triggers Claude Code's Ink framework clipboard check. (5) Server polls `xclip -t TARGETS -o` to confirm clipboard ready before responding.
- Dead dtach sockets cause "connection refused": `restoreSessions()` tried to attach to dead sockets. Fix: verify via `fuser`/`pgrep` before attach, auto-clean dead sockets.
- node-pty `posix_spawnp failed` on macOS: prebuilt binary incompatible, `build/Release/spawn-helper` missing. Fix: `npm rebuild node-pty --build-from-source`. Also: commands (dtach, node, env, claude) resolved to full paths at startup via `resolveCmd()` since node-pty's `posix_spawnp` may not find Homebrew paths.
- File explorer path not restored: `_explorerPath` was set via monkey-patch of `navigate()` AFTER constructor, so `_loadHome()`'s navigate used the original unpatched version. Fix: set `winInfo._explorerPath` directly inside `FileExplorer.navigate()`. Also pass `startPath` to constructor to skip `_loadHome()` when restoring.
